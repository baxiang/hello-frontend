# 类型注解初体验 ⭐

> 从 JavaScript 的问题出发，理解 TypeScript 的类型注解

---

## 学习目标

学完本节，你能：
- 理解为什么 JavaScript 需要类型注解
- 写出基本的类型注解代码
- 看懂 TypeScript 的编译错误并修复

## 生活化比喻

**类型注解就像"给仓库的箱子贴标签"**：

```
没有标签的仓库（JavaScript）：
┌─────────────────────────────┐
│  📦  📦  📦  📦            │
│  不知道里面是什么           │
│  用的时候才发现拿错了       │
└─────────────────────────────┘

贴好标签的仓库（TypeScript）：
┌─────────────────────────────┐
│  📦文字  📦数字  📦布尔    │
│  拿之前就知道里面是什么     │
│  拿错了在门口就被拦下       │
└─────────────────────────────┘
```

## 第一步：看看 JavaScript 的问题

先来看一段 JavaScript 代码：

```javascript
function add(a, b) {
    return a + b;
}
```

这个函数看起来没问题，对吧？我们来调用一下：

```javascript
add(1, 2);       // 返回 3 ✅ 正常
add('1', '2');   // 返回 '12' ⚠️ 字符串拼接了！
add(1, 'hello'); // 返回 '1hello' ❌ 这肯定不是你想要的
```

**发现问题了吗？**

JavaScript 不会阻止你传错误类型的参数，只有运行的时候才会出现奇怪的结果。如果这个函数在一个大型项目里，你可能要调试很久才能找到问题。

---

## 第二步：TypeScript 怎么解决？

TypeScript 的做法很简单——**给参数加上类型标签**：

```typescript
function add(a: number, b: number): number {
    return a + b;
}
```

看到 `: number` 了吗？这就是**类型注解**。它告诉 TypeScript：
- `a: number` → 参数 a 必须是数字
- `b: number` → 参数 b 必须是数字
- `: number`（最后那个）→ 返回值也是数字

现在如果你传错了类型：

```typescript
add(1, 2);       // ✅ 通过
add('1', '2');   // ❌ 编译报错：类型"string"不能赋给类型"number"
```

**关键区别：**
- JavaScript：运行时报错（用户发现了）
- TypeScript：编译时报错（你自己就发现了）

## 第三步：试试基础类型

TypeScript 有 3 种最基础的类型：

### string（字符串）

就是文字：

```typescript
let name: string = '张三';
let greeting: string = '你好，' + name;
```

### number（数字）

整数和小数都是 number：

```typescript
let age: number = 25;
let price: number = 19.99;
```

### boolean（布尔值）

只有两个值：`true` 或 `false`：

```typescript
let isActive: boolean = true;
let isDeleted: boolean = false;
```

## 第四步：数组和对象

### 数组

数组的类型注解有两种写法：

```typescript
// 写法 1：类型后面加 []
let numbers: number[] = [1, 2, 3];

// 写法 2：Array<类型>
let names: Array<string> = ['Tom', 'Jerry'];
```

两种写法完全一样，第一种更常用。

### 对象

对象需要描述每个属性的类型：

```typescript
let user: { name: string; age: number } = {
    name: '张三',
    age: 25
};
```

这看起来有点长对吧？没关系，后面我们会学用 `interface` 简化它。

---

## 第五步：TypeScript 其实很聪明（类型推断）

好消息是——**你不需要每次都写类型注解**。

TypeScript 会根据你赋的值自动推断类型：

```typescript
// 你不写，TypeScript 也知道 name 是 string
let name = '张三';  // 推断为 string

// 它也知道 age 是 number
let age = 25;       // 推断为 number

// 它甚至知道数组元素的类型
let numbers = [1, 2, 3];  // 推断为 number[]
```

**所以什么时候需要手动写类型注解呢？**

1. **函数参数** — TypeScript 猜不到：
   ```typescript
   function greet(name: string) { }  // 必须写
   ```

2. **空数组/空对象** — TypeScript 不知道里面会放什么：
   ```typescript
   let numbers: number[] = [];  // 必须写
   ```

3. **函数返回值** — 简单时可以省略，复杂时建议写：
   ```typescript
   function add(a: number, b: number) {
       return a + b;  // TypeScript 推断返回 number
   }
   ```

## 第六步：当类型不匹配时

如果你写了类型注解，但赋的值类型不对，TypeScript 会报错：

```typescript
let age: number = '25';
// ❌ 报错：类型 "string" 不能赋给类型 "number"
```

这个报错是在**编译时**出现的，也就是说你的代码还没运行就已经发现问题了。

### 常见错误对照表

| 错误信息 | 什么意思 | 怎么修 |
|---------|---------|--------|
| `Type 'string' is not assignable to type 'number'` | 你给数字类型赋了字符串 | 检查赋值，改成正确的类型 |
| `Property 'x' does not exist on type` | 你访问了不存在的属性 | 检查类型定义，看看属性名对不对 |
| `Parameter 'x' implicitly has an 'any' type` | 函数参数缺少类型标注 | 加上 `x: number` |

---

## 第七步：一个不推荐的类型 — any

TypeScript 有一个特殊类型叫 `any`，意思是"什么类型都行"：

```typescript
let data: any = 'hello';
data = 42;       // ✅ 可以
data = true;     // ✅ 也可以
```

**为什么不推荐？**

因为 `any` 等于关闭了类型检查：

```typescript
function process(data: any) {
    console.log(data.name);  // 编译通过，但运行时可能报错
}
```

**正确做法：定义具体类型**

```typescript
interface Data { name: string }
function process(data: Data) {
    console.log(data.name);  // 有完整的类型提示
}
```

> 💡 **建议**：学习阶段可以暂时用 `any`，但尽量早点改掉这个习惯。

## 总结

| 概念 | 语法 | 示例 |
|------|------|------|
| 字符串 | `: string` | `let name: string = '张三'` |
| 数字 | `: number` | `let age: number = 25` |
| 布尔值 | `: boolean` | `let active: boolean = true` |
| 数组 | `: 类型[]` | `let nums: number[] = [1, 2]` |
| 对象 | `: { 属性: 类型 }` | `let u: { name: string } = { name: '张三' }` |
| 函数参数 | `参数名: 类型` | `function fn(name: string)` |
| 函数返回值 | `): 类型` | `function fn(): number` |

**记住：**
- TypeScript 会帮你推断大部分类型
- 函数参数必须写类型注解
- 不要用 `any`，定义具体类型

## 常见问题

### Q：TypeScript 的类型注解会不会让代码变慢？

不会。类型注解只在编译时检查，编译后的 JavaScript 代码里没有任何类型信息。

### Q：我可以用 JavaScript 写项目吗？

可以，但大型项目用 TypeScript 能减少很多 bug。建议新项目直接用 TypeScript。

### Q：什么时候用 `any`？

学习阶段或者迁移老项目时可以暂时用，但应该尽快替换为具体类型。
