# 技术内容编写规范

本文档定义前端学习教程的技术内容编写标准，目标是让开发者从入门成长为前端高级技术专家。

## 经典书籍的写作特点借鉴

> 学习《JavaScript高级程序设计》、《你不知道的JavaScript》等经典书籍的写作方法

### 经典书籍为什么经典不衰

| 书籍 | 核心优势 | 我们要借鉴什么 |
|------|---------|---------------|
| **《JavaScript高级程序设计》** | 全面、循序渐进、实战导向 | 知识体系的完整性、从基础到进阶的过渡 |
| **《你不知道的JavaScript》** | 深挖原理、直面误区、易懂解释 | 不停留在表面、揭示底层机制 |

| **《Eloquent JavaScript》** | 最简示例、边学边练、理论实践结合 | 从最简代码开始，逐步扩展 |

### 生活化比喻：降低理解门槛

**经典书籍做法：** 用日常熟悉场景类比技术概念

| 技术概念 | 生活化比喻 | 说明 |
|---------|-----------|------|
| **作用域** | 楼层的玻璃窗 | 每层只能看到本层和下层，看不到上层 |
| **闭包** | 带着行李离开酒店 | 函数执行完，但携带的变量不消失 |
| **原型链** | 家族遗传谱 | 子类继承父类特征，父类继承祖先特征 |
| **事件循环** | 餐厅点餐流程 | 同步点餐 → 异步做菜 → 按顺序上菜 |
| **Promise** | 预订电影票 | 先拿凭证，电影开场时兑换 |
| **TypeScript类型** | 餐厅菜单标注 | 菜单说明每道菜的成分，避免过敏 |

**比喻写作规范：**

```markdown
### 概念名称

**生活化比喻：** 用一句话类比日常场景。

例如：闭包就像"带着行李离开酒店"——函数执行完毕（离开酒店），但内部变量（行李）不消失。

**技术定义：** 一句话技术定义。

**比喻对应：**

| 比喻元素 | 技术对应 |
|---------|---------|

```

### 最简示例优先：从简单到复杂

**经典书籍做法：** 先展示 1-3 行最简代码，再逐步扩展

```markdown
错误做法：直接展示复杂代码
```javascript
// 50行的复杂示例，读者看不懂
...
```

正确做法：最简示例 → 详细示例 → 实际应用
```javascript
// 1. 最简示例（1-3行）
const arr = [1, 2, 3];
arr.map(x => x * 2);  // [2, 4, 6]

// 2. 详细示例（5-10行）
const users = [
  { name: '张三', age: 20 },
  { name: '李四', age: 25 }
];
users.map(user => user.name);  // ['张三', '李四']

// 3. 实际应用（完整场景）
// 在 React 组件中使用
function UserList({ users }) {
  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
```
```

### 底层原理示例

**示例：JavaScript 事件循环**

```markdown
### L3 专家层：底层原理

#### JavaScript 如何实现

事件循环是 JavaScript 异步的核心机制：

```
事件循环结构：
┌──────────────────────────────────────────────────┐
│                   调用栈 (Call Stack)              │
│  [函数调用 → 执行 → 弹出]                         │
└──────────────────────────────────────────────────┘
          ↓ (同步任务完成)
┌──────────────────────────────────────────────────┐
│                   微任务队列 (Microtask)           │
│  Promise.then、queueMicrotask                     │
│  → 栈空时立即执行全部                             │
└──────────────────────────────────────────────────┘
          ↓ (微任务清空)
┌──────────────────────────────────────────────────┐
│                   宏任务队列 (Macrotask)           │
│  setTimeout、setInterval、I/O                     │
│  → 每次只执行一个                                 │
└──────────────────────────────────────────────────┘
          ↓ (取一个宏任务)
       ┌─────────────┐
       │  事件循环    │ ← 循环往复
       └─────────────┘
```

```javascript
// 演示事件循环执行顺序
console.log('1. 同步开始');

setTimeout(() => console.log('4. 宏任务'), 0);

Promise.resolve().then(() => console.log('3. 微任务'));

console.log('2. 同步结束');

// 输出顺序：1 → 2 → 3 → 4
// 同步代码 → 微任务 → 宏任务
```

#### 性能考量

| 操作 | 执行时机 | 说明 |
|------|---------|------|
| 同步代码 | 立即 | 阻塞主线程 |
| 微任务 | 同步后、宏任务前 | 执行完所有才继续 |
| 宏任务 | 每轮循环执行一个 | 适合延迟执行 |

**注意：** 微任务过多会阻塞宏任务，导致页面卡顿。

#### 设计动机

JavaScript 单线程设计的原因：

| 设计选择 | 原因 | 影响 |
|----------|------|------|
| 单线程 | 避免锁、竞态条件 | 无并发安全问题 |
| 事件循环 | 处理异步不阻塞 | UI 响应流畅 |
| 微任务优先 | Promise 需要尽快执行 | 异步链式调用更高效 |

#### 知识关联

```
事件循环关联：
┌───────────────────┐     ┌───────────────────┐
│    同步/异步      │────→│    事件循环       │
└───────────────────┘     └───────────────────┘
                              │
          ┌───────────────────┼───────────────────┐
          ↓                   ↓                   ↓
┌───────────────────┐ ┌───────────────────┐ ┌───────────────────┐
│    Promise        │ │   setTimeout      │ │    async/await    │
└───────────────────┘ └───────────────────┘ └───────────────────┘
```
```

## 完整示例：CSS Flexbox

```markdown
### Flexbox 弹性布局

**定义：** 一维弹性容器布局模式，自动处理子元素排列、对齐、分配空间。

#### L1 理解层：语法结构

```
Flexbox 结构：
┌──────────────────────────────────────────────────────┐
│  display: flex;                                       │
│                                                       │
│  主轴方向 ──────────────────────────────────────→    │
│                                                       │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐                     │
│  │ 子1 │ │ 子2 │ │ 子3 │ │ 子4 │                     │
│  └─────┘ └─────┘ └─────┘ └─────┘                     │
│                                                       │
│  justify-content: 主轴对齐                            │
│  align-items:     交叉轴对齐                          │
│  flex-direction:  主轴方向（row/column）              │
└──────────────────────────────────────────────────────┘
```

**最简示例：**

```html
<style>
.container {
  display: flex;            /* 启用 Flexbox */
  justify-content: center;  /* 主轴居中 */
  align-items: center;      /* 交叉轴居中 */
}
</style>

<div class="container">
  <div class="item">居中元素</div>
</div>
```

**详细示例：**

```html
<style>
.navbar {
  display: flex;
  justify-content: space-between;  /* 两端对齐 */
  align-items: center;
  gap: 1rem;                        /* 子元素间距 */
}

.navbar .logo {
  flex: 0 0 auto;                   /* 不伸缩 */
}

.navbar .links {
  flex: 1;                          /* 占据剩余空间 */
  display: flex;
  gap: 2rem;
}
</style>

<nav class="navbar">
  <div class="logo">Logo</div>
  <div class="links">
    <a href="#">首页</a>
    <a href="#">产品</a>
    <a href="#">关于</a>
  </div>
</nav>
```

**关键属性说明：**

| 属性 | 含义 | 常用值 |
|------|------|--------|
| `display: flex` | 启用 Flexbox | `flex`, `inline-flex` |
| `flex-direction` | 主轴方向 | `row`, `column` |
| `justify-content` | 主轴对齐 | `center`, `space-between` |
| `align-items` | 交叉轴对齐 | `center`, `stretch` |
| `gap` | 子元素间距 | `1rem`, `10px 20px` |

#### L2 实践层：最佳实践

**推荐做法：**

| 做法 | 原因 | 示例 |
|------|------|------|
| 用 Flexbox 做一维布局 | 语义清晰，代码简洁 | 导航栏、卡片列表 |
| 用 Grid 做二维布局 | 同时控制行列 | 网格系统、复杂布局 |
| `gap` 替代 margin | 间距统一，无需负 margin | `gap: 1rem` |
| `flex: 1` 替代 `flex-grow: 1` | 简写更清晰 | 自适应宽度 |

**反模式：不要这样做：**

```html
<!-- ❌ 用多个 Flexbox 嵌套做二维布局 -->
<style>
.wrapper {
  display: flex;
  flex-direction: column;
}
.row {
  display: flex;
}
</style>

<!-- 问题：
  1. 嵌套层级多，难以维护
  2. 无法控制列宽统一
  3. Grid 可以一行搞定 -->
```

```html
<!-- ✅ 正确做法：Grid 做二维布局 -->
<style>
.grid {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 1rem;
}
</style>
```

**适用场景：**

| 场景 | 是否推荐 | 原因 |
|------|---------|------|
| 导航栏 | ✅ 推荐 | 一行排列，两端对齐 |
| 卡片列表 | ✅ 推荐 | 自动换行（flex-wrap） |
| 居中元素 | ✅ 推荐 | 最简洁方案 |
| 网格系统（行列控制） | ❌ 不推荐 | 用 Grid |
| 复杂二维布局 | ❌ 不推荐 | 用 Grid |

#### L3 专家层：底层原理

**浏览器如何实现：**

Flexbox 布局计算分两步：

```
Flexbox 计算流程：
┌──────────────────────────────────────────────────┐
│  步骤 1：收集子元素信息                           │
│  - 基础尺寸（content、width）                     │
│  - flex-grow、flex-shrink                        │
│  - min/max-width 限制                            │
└──────────────────────────────────────────────────┘
          ↓
┌──────────────────────────────────────────────────┐
│  步骤 2：分配空间                                 │
│  - 计算剩余/不足空间                              │
│  - 按 flex-grow 比例放大                          │
│  - 按 flex-shrink 比例缩小                        │
│  - 应用对齐（justify-content、align-items）       │
└──────────────────────────────────────────────────┘
```

**性能考量：**

| 操作 | 性能影响 | 说明 |
|------|---------|------|
| 初始 Flexbox | 中等 | 触发一次布局计算 |
| 子元素数量变化 | 高 | 重新计算所有子元素空间 |
| flex 属性变化 | 高 | 触发重排 |
| 对齐属性变化 | 低 | 只影响位置，不改变尺寸 |

**设计动机：**

| 设计选择 | 原因 | 影响 |
|----------|------|------|
| 一维设计 | 简化概念，降低门槛 | 与 Grid（二维）互补 |
| flex-grow/shrink | 自适应空间分配 | 响应式更简单 |
| gap 属性 | 统一间距语义 | 替代复杂的 margin 方案 |

**知识关联：**

```
Flexbox 关联：
┌───────────────────┐     ┌───────────────────┐
│     Block 布局    │     │   Flexbox 布局    │
└───────────────────┘     └───────────────────┘
                              │
                              ↓
                        ┌───────────────────┐
                        │     Grid 布局     │
                        │   （二维扩展）    │
                        └───────────────────┘
                              │
          ┌───────────────────┼───────────────────┐
          ↓                   ↓                   ↓
┌───────────────────┐ ┌───────────────────┐ ┌───────────────────┐
│  flex-direction   │ │  justify-content  │ │    align-items    │
└───────────────────┘ └───────────────────┘ └───────────────────┘
```

### 反模式清单

教程代码必须避免以下反模式：

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| 内联样式过多 | 难维护、无复用 | 用 CSS 类 |
| `var` 声明变量 | 变量提升、作用域混乱 | 用 `const`/`let` |
| 直接操作 DOM | React/Vue 不推荐 | 用状态驱动 |
| 全局变量 | 隐式依赖 | 用模块/状态管理 |
| 过深嵌套回调 | 可读性差（回调地狱） | 用 async/await |
| 不处理错误 | 用户无感知 | try-catch 或 .catch() |

```javascript
// ❌ 回调地狱
fetchData(function(a) {
  fetchMore(a, function(b) {
    fetchMore(b, function(c) {
      // ...
    });
  });
});

// ✅ async/await
async function getAllData() {
  const a = await fetchData();
  const b = await fetchMore(a);
  const c = await fetchMore(b);
  return c;
}
```

```javascript
// ❌ 不处理错误
fetch('/api/data').then(res => res.json()).then(data => {
  console.log(data);
});

// ✅ 处理错误
try {
  const res = await fetch('/api/data');
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  const data = await res.json();
  console.log(data);
} catch (error) {
  console.error('获取数据失败:', error);
}
